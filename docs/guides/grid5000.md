# Production builds on Grid'5000

This guide covers the supported bounded streaming workflow. Sentence
segmentation runs only inside an allocated CUDA job. Frontends prepare the
checkout, submit jobs, and inspect durable logs; they never construct the SaT
model or run inference.

## Prerequisites and storage model

You need a Grid'5000 account, access to a GPU site, an existing Hugging Face
output dataset, and standard Hub authentication in the remote environment.
The scripts never accept a token argument.

Keep the Git checkout, virtual environment, Hugging Face model cache, and logs
on persistent storage. Allocation-local scratch holds only the current input
shard and temporary output. Completed shard checkpoints are verified on the
Hub before local scratch is evicted, so another allocation can resume without
repeating completed work. Never place a dataset mirror on a frontend `/tmp`.

## Immutable revisions

Pin both identities before submission:

```bash
export INPUT_REPO_ID="NoeFlandre/osm-polygon-wikidata-only"
export OUTPUT_REPO_ID="NoeFlandre/osm-polygon-wikidata-sentence-relevance"
export INPUT_REVISION="<40-character-input-commit>"
export SOURCE_COMMIT="$(git rev-parse HEAD)"
test "$(git rev-parse origin/main)" = "${SOURCE_COMMIT}"
```

`INPUT_REVISION` and `SOURCE_COMMIT` must be lowercase 40-character Git
identifiers. Mutable names such as `main` are rejected by the submission
scripts.

## Frontend preparation

Define persistent directories and install the locked environment from the
clean detached checkout:

```bash
export REPO_ROOT="${HOME}/g5k-sentence-build/repo"
export HF_HOME="${HOME}/g5k-sentence-build/hf-home"
export LOG_ROOT="${HOME}/g5k-sentence-build/logs"
mkdir -p -m 0700 "${HF_HOME}" "${LOG_ROOT}"
git -C "${REPO_ROOT}" status --porcelain
uv sync --locked --extra hub --extra segmentation -C "${REPO_ROOT}"
```

The status command must print nothing. Model files may be cached in
`HF_HOME`; input Parquet files are fetched one shard at a time by the compute
job.

## Bounded streaming submission

Choose a stable run ID. `MAX_SHARDS=0` processes every remaining shard;
positive values are useful for a canary. The only canonical build submission
command is:

```bash
"${REPO_ROOT}/scripts/grid5000/submit_streaming_build.sh" \
  "${REPO_ROOT}" "${HF_HOME}" "${LOG_ROOT}" \
  "${OUTPUT_REPO_ID}" "${INPUT_REPO_ID}" \
  "${SOURCE_COMMIT}" "${INPUT_REVISION}" \
  "sentence-build-v1" "128" "0"
```

The adapter submits one non-interactive production job requesting one GPU.
The compute wrapper verifies the checkout, runs the CUDA preflight, and invokes
`scripts.streaming.driver` with `--device cuda`. It does not fall back to CPU,
MPS, or automatic device selection. The driver downloads at most one shard's
six required Parquet inputs, processes it, uploads and independently verifies
the checkpoint, then evicts local shard data.

## Monitoring and resume

Capture the numeric job ID printed by OAR and monitor it from the frontend:

```bash
oarstat -j "${OAR_JOB_ID}"
find "${LOG_ROOT}/${OAR_JOB_ID}" -maxdepth 1 -type f -print
```

Inspect `build.exit_code`, `gpu_preflight.json`, and the stdout/stderr logs.
Do not edit a checkpoint branch manually. To resume after walltime or a
transient failure, submit the same command with the same run ID, source commit,
input revision, model, pipeline version, and batch size. Verified remote
checkpoints are reused; mismatched identities fail closed.

## Deterministic finalization

Finalization is a separate bounded job. For a single-region release, set
`EXPECTED_SHARD` to its canonical shard key and use the checkpoint branch
created by the build:

```bash
export RUN_ID="sentence-build-v1"
export STAGING_REVISION="checkpoints/${RUN_ID}"
export EXPECTED_SHARD="afghanistan-latest"
"${REPO_ROOT}/scripts/grid5000/submit_streaming_finalization.sh" \
  "${REPO_ROOT}" "${HF_HOME}" "${LOG_ROOT}" \
  "${OUTPUT_REPO_ID}" "${INPUT_REPO_ID}" \
  "${SOURCE_COMMIT}" "${INPUT_REVISION}" \
  "${RUN_ID}" "${STAGING_REVISION}" "${EXPECTED_SHARD}" \
  "02:00:00" "cpu"
```

The job streams verified checkpoints, performs global deterministic
finalization, and persists `sentences.parquet`, `manifest.json`, and
`README.md` beneath `${LOG_ROOT}/${OAR_JOB_ID}/output/` before allocation
scratch is removed. GPU finalization is supported only when explicitly chosen;
it is normally unnecessary.

## Validation and publication boundary

Validate the persistent output before any publication:

```bash
uv run python - <<'PY'
from pathlib import Path
from osm_polygon_sentence_relevance.output.validation import validate_export_directory

validate_export_directory(Path("/absolute/path/to/job/output"))
print("validation passed")
PY
```

Publication is a separate remote mutation. Publish only a complete, validated
release, then independently download it and validate the readback. The
publisher targets an existing repository, uses one Hub commit, accepts no token
argument, and never creates repositories.

## Failure recovery and cleanup

- A non-zero job exit preserves logs and any verified remote checkpoints.
- Reuse the same run identity to resume; use a new run ID for different inputs
  or code.
- Never run inference on a frontend or local Mac as a fallback.
- Remove allocation scratch only through the guarded job wrapper. Delete old
  checkpoint branches only after the final release has been validated and the
  deletion has been explicitly approved.
- Persistent caches and logs may be removed when no longer needed; never delete
  an active job's directory.
