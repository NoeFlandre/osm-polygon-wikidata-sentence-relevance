# Grid'5000 CUDA smoke test (Phase 9B)

This guide covers the minimal, credential-free operational tooling that
proves the real `wtpsplit` SaT model constructs and performs
multilingual inference on a remote Grid'5000 NVIDIA GPU — **not** on
the MacBook and **not** on a compute-node CPU.

It is a *smoke test* only. It is deliberately scoped to:

- the reusable operational scripts under `scripts/grid5000/`;
- a one-GPU, ≤ 30-minute OAR allocation;
- a small multilingual batch (English, French, one non-Latin script);
- a compact JSON result written to *persistent* storage you choose.

It does **not** add multi-GPU execution, distributed workers, a full
dataset build, automatic publishing, automatic repository creation,
token handling, or a remote service.

## Roles: Mac vs Grid'5000

- **The Mac is used only for local tests, staging, and asset
  preparation.** All `pytest`, `ruff`, `mypy`, and packaging run
  here. No SaT model is constructed and no inference is performed
  on the Mac.
- **Real SaT inference runs inside an allocated OAR CUDA
  allocation on a Grid'5000 compute node.** The model is
  placed on `cuda` explicitly; the segmenter verifies the placement
  and would fail rather than silently fall back to CPU.

Acceptance criterion: the smoke log must show
`resolved_device == "cuda"` and a completed multilingual inference.
**CPU fallback is forbidden** — if CUDA placement or inference
fails, the run stops; it is never retried on CPU, MPS, or the
Mac.

## Access (Nancy, placeholders)

Direct SSH to a site login node (`<login>@nancy.grid5000.fr`) is **not**
generally valid from outside the Grid'5000 network; it requires
ProxyJump configuration. Use either of the two patterns below, and
substitute your own Grid'5000 login. No personal account is mentioned
in this repository; treat every concrete `<login>` here as a
placeholder.

**Option A — chained SSH through the frontend:**

```bash
ssh <your-login>@access.grid5000.fr          # Grid'5000 frontend
ssh nancy                                   # Nancy site
```

**Option B — a local `~/.ssh/config` alias using `ProxyJump`:**

```text
# ~/.ssh/config (do not commit)
Host access.grid5000.fr
    User <your-login>

Host nancy
    HostName nancy.grid5000.fr
    User <your-login>
    ProxyJump access.grid5000.fr
```

With option B, `ssh nancy` from the local shell reaches the Nancy
site in one step.

OAR jobs are submitted from the frontend (or a site login node) with
`oarsub`. The smoke payload itself is **not** a submission tool;
it is run *inside* the allocation after the scheduler has set
`OAR_JOB_ID`.

`CUDA_VISIBLE_DEVICES` is **informational only** on Grid'5000.
Grid'5000 scopes reserved GPUs through its resource isolation and
does not guarantee that `CUDA_VISIBLE_DEVICES` is exported on a
given allocation. The smoke payload therefore never requires,
reads, assigns, defaults, normalises, prints, or exports it. If
the scheduler happens to set it, the harness inherits it
unchanged. The authoritative runtime proof of GPU scoping is
`torch.cuda.device_count() == 1` inside `gpu_preflight.py`, which
runs as the first phase of the smoke. See the [official Grid'5000
GPU reservation
documentation](https://www.grid5000.fr/w/GPUs_on_Grid5000) for the
scheduler-level reservation model.

## OAR allocation (smoke)

The smoke is dispatched through two complementary flows:

### Phase 9D/9F: non-interactive batch submission (canonical)

For automation and unattended smoke runs, the canonical
submission uses a two-stage design. The scheduler — not a held SSH
shell — owns the allocation lifetime, which removes the interactive
`-I` frag failure mode.

**Stage 1 (frontend).** Run the submission helper directly on the
frontend. It validates the four arguments, serialises them into the
single command string this OAR version requires, and invokes
`oarsub` exactly once. The submission helper is **not** the OAR
batch payload; the helper only builds and submits the job.

```bash
"${REPO_ROOT}/scripts/grid5000/submit_gpu_smoke.sh" \
  "${REPO_ROOT}" \
  "${HF_HOME}" \
  "${LOG_ROOT}" \
  "${SOURCE_COMMIT}"
```

**Stage 2 (compute node).** Inside the allocation, the helper's
command string runs the compute-node wrapper
`scripts/grid5000/run_gpu_smoke_job.sh`, which invokes the smoke
harness and writes the six-artifact bundle. OAR records stdout,
stderr, the exit status, and the resource assignment without
relying on a TTY-bridge.

**WARNING:** the canonical frontend form must **not** be prefixed
by `oarsub` — the helper performs the single `oarsub` invocation
itself, and this OAR version rejects `oarsub ... wrapper arg1 arg2
arg3 arg4`. The four variables must already be exported in the
frontend shell (see *Persistent storage layout* below); quoting
each preserves spaces in paths and protects against
shell-metacharacter injection.

The canonical request guarantees:

- exactly one GPU;
- maximum 30-minute walltime;
- the production queue (no preemptible `besteffort`);
- no unnecessary `allow_classic_ssh` job type;
- no unverified resource-property syntax (e.g.
  `-p "gpu_model='L40S'"`).

The compute-node wrapper, run inside the allocation via the
submission helper, requires the four explicit positional arguments
(`REPO_ROOT`, `HF_HOME`, `LOG_ROOT`, `EXPECTED_SOURCE_COMMIT`) so
the smoke is *self-describing* and does not depend on a frontend
shell to thread values through. The wrapper does **not** verify
CUDA placement itself; that remains the compute-node preflight in
`run_gpu_smoke.sh`. The submission helper does **not** poll,
cancel, or retry; if the job lingers, the operator inspects it
explicitly with `oarstat -j <job_id>` and decides on next actions.

### Interactive smoke (human-held TTY only)

The earlier interactive form is supported only when a human holds
the TTY-bridge explicitly:

```bash
oarsub -q production -l gpu=1,walltime=00:30:00 -I
```

This is suitable for interactive debugging on a remote terminal.
It is *not* suitable for automation: the local shell can be
killed by tool timeouts, network drops, or laptop sleeps, and OAR
interprets the empty interactive shell as a frag request and
finalizes the allocation without running any payload. The Phase
9D batch submission is therefore canonical; the interactive form
exists only to assist in manual triage.

The smoke payload itself verifies the actual GPU and CUDA runtime
through `gpu_preflight.py`; the OAR command does not need to
hard-code a GPU family. If a GPU-family constraint is later
desired, first inspect Nancy's exact OAR property name and value
read-only (for example `oarnodes -p gpu_model -h gres-1.nancy.grid5000.fr`)
and test the request syntax on an idle production queue before
making it part of the documented plan. Do not guess.

The earlier draft of this guide proposed the unverified command
`-p "gpu_model='L40S'" -t besteffort -t allow_classic_ssh`. **Do
not use that command.** It was never executed and the resource
property syntax in particular was not validated against the live
Nancy OAR schema.

**Phase 9H reminder:** when the interactive form above is used,
the scheduler may or may not set `CUDA_VISIBLE_DEVICES`. The
smoke payload itself does not depend on this variable; the
preflight's `torch.cuda.device_count() == 1` check is the
authoritative runtime proof of GPU scoping.

## Commit-based remote reproducibility

The remote smoke must be executed against an exact, pushed commit.
The Mac must not rsync an uncommitted source tree for execution.

Required sequence:

1. Each amendment lands as a separate, fully reviewed commit
   on `main`, pushed only after explicit approval.
2. The operator exports the exact pushed source commit as
   `SOURCE_COMMIT` and re-uses it for every checkout and
   verification step in this guide.
3. On the remote site, the agent pins the checkout to the
   exact `SOURCE_COMMIT`:
   `git checkout --detach "${SOURCE_COMMIT}"`.
4. `git -C "${REPO_ROOT}" status --porcelain` must be empty
   before environment setup and before the smoke is launched.
   No modified, staged, or untracked files in the working
   tree.
5. The ${SOURCE_COMMIT} SHA is recorded in the remote smoke
   report (`run_metadata.json`) so the exact source tree used
   for the smoke can be re-derived later.
6. The only rsync/download transfer is for model and cache
   assets, never for modified source code or lockfiles. In other
   words, `uv sync --locked` must run from a clean checkout on
   the compute node — not from a tarball that bundles local
   edits.

If `git -C "${REPO_ROOT}" status --porcelain` reports any
change on the remote checkout, **stop**: clean the tree (or
re-checkout `SOURCE_COMMIT`) and restart from the
environment-setup step. Do not proceed with a dirty tree.

## One-GPU scope

Request exactly **one** GPU. Multi-GPU is not implemented and is not
needed for the smoke. A short walltime is sufficient: cap the first
smoke at **30 minutes** (`-l walltime=0:30:00`). Let OAR pick
the specific resource; do not hard-code a particular node.

The preflight script refuses to start if `torch.cuda.device_count()`
reports anything other than exactly **1** visible device. This guards
against an incorrectly scoped OAR allocation or missing CUDA
visibility isolation; the smoke cannot run on a CPU fallback or a
multi-GPU allocation. `CUDA_VISIBLE_DEVICES` is not consulted by the
preflight: Grid'5000 does not guarantee it is exported, and the
authoritative runtime proof of GPU scoping is the
`torch.cuda.device_count() == 1` check combined with
`torch.cuda.get_device_name(0)` succeeding. The preflight does not
read or require `CUDA_VISIBLE_DEVICES`; see
[the Grid'5000 GPU reservation
documentation](https://www.grid5000.fr/w/GPUs_on_Grid5000) for the
scheduler-level reservation model.

## Locked interpreter

The smoke payload never invokes `python`, `python3`, `uv run`, or
activates a shell environment. Instead it requires the
**locked interpreter** at `${REPO_ROOT}/.venv/bin/python`, created
*beforehand* on the compute node with:

```bash
cd <your-repo-checkout>
uv sync --locked --extra segmentation
```

Both the preflight and the real SaT inference are invoked through
this exact interpreter. The preflight and the smoke itself refuse to
start unless the interpreter exists as a regular, executable file.
The job payload does **not** install, resolve, or update
dependencies; that work is part of the environment-setup step
above, run once before submitting the OAR job.

## Recommended Nancy GPU families

The following families are suitable for the smoke (availability varies
by queue and reservation):

- **L40S** — efficient, single-GPU friendly.
- **A40** — broadly available compute GPU.
- **A100** — high-memory option if a reservation exposes it.

The preflight script reports the actual `device_0_name`, so the log
establishes which GPU satisfied the run.

## `CUDA_VISIBLE_DEVICES` is informational only

The smoke payload — wrapper, harness, and preflight alike — does
**not** treat `CUDA_VISIBLE_DEVICES` as a hard contract:

- Grid'5000 scopes reserved GPUs through OAR's resource isolation
  (`-l gpu=1`). The scheduler does **not** guarantee
  `CUDA_VISIBLE_DEVICES` is exported on a given allocation. See
  [the Grid'5000 GPU reservation
  documentation](https://www.grid5000.fr/w/GPUs_on_Grid5000).
- The compute-node wrapper (`run_gpu_smoke_job.sh`) never reads,
  assigns, defaults, normalises, prints, or exports the variable.
- The smoke harness (`run_gpu_smoke.sh`) never reads, requires,
  assigns, defaults, normalises, prints, or exports the variable.
- The preflight (`gpu_preflight.py`) deliberately does **not** read
  the variable; the authoritative runtime proof of GPU scoping is
  the combination of `torch.cuda.is_available() is True`,
  `torch.cuda.device_count() == 1`, and
  `torch.cuda.get_device_name(0)` succeeding. The preflight's
  result JSON does not include a `cuda_visible_devices` field.

If the scheduler happens to set `CUDA_VISIBLE_DEVICES`, the
compute-node wrapper inherits it unchanged (the wrapper's
`bash "${SMOKE_HARNESS}"` invocation does not touch the
environment), and the smoke harness inherits it unchanged
likewise. The smoke neither reads it nor rewrites it. The
documentation, scripts, and tests assert this contract together:
no assignment, no default, no normalisation, no export, no print.

## Persistent storage requirement

Grid'5000 compute nodes are **ephemeral**; local scratch is lost
when the job ends. You must select **persistent** paths for:

- `HF_HOME` — the Hugging Face cache (model + tokenizer). Stage
  `sat-3l-sm` and its XLM-R cache here *before* the job, or
  reuse an already-staged cache.
- `SMOKE_LOG_DIR` — where the compact JSON result is written.

Do **not** hard-code `/home`, `/srv/storage`, `/tmp`, or any
personal absolute path in the scripts. Both `HF_HOME` and
`SMOKE_LOG_DIR` are **required caller-provided inputs**. Example
shape (yours will differ; use your own persistent directory):

```bash
export HF_HOME=/path/to/your/persistent/hf_home
export SMOKE_LOG_DIR=/path/to/your/persistent/smoke_logs
```

## Quota checks

Home and persistent storage have soft/hard quotas. Check before
staging assets:

```bash
quota -s                 # home / persistent quota
df -h /path/to/your/persistent
```

The previously observed footprint for this account was approximately
**55.9 GB used** against a **24.4 GB soft quota** and a
**97.7 GB hard limit** — i.e. already over the soft limit. Do
not assume `$HOME` is a safe stage; prefer a dedicated persistent
directory and verify headroom before copying model caches.

## Offline asset staging

The smoke runs with `HF_HUB_OFFLINE=1` and
`TRANSFORMERS_OFFLINE=1`: it will **not** fetch from the
network at runtime. Therefore the required assets must already be
present in `HF_HOME`:

- `sat-3l-sm` (the `wtpsplit` SaT weights), and
- the XLM-R base/backbone cache it depends on.

Public Hugging Face assets like `sat-3l-sm` and the XLM-R base do
**not** require authentication; they can be downloaded by the
`huggingface_hub` library on any node with general outbound HTTPS
access. To populate `HF_HOME` without performing inference, the
preferred workflow is a *download-only* script that uses
`huggingface_hub.snapshot_download` (or the equivalent `hf` CLI) to
fetch the files into `HF_HOME` without ever instantiating the model.

**Forbidden during cache staging:**

- No model construction on the Mac or the Grid'5000 frontend.
- No inference on the Mac or the Grid'5000 frontend.
- No transfer of the entire Mac Hugging Face cache by default —
  prefer fetching only the immutable revisions the smoke needs.
- No `HF_TOKEN` unless a particular asset demonstrably requires
  it; tokens must never be logged.

Concrete transfer/download options:

1. **Download-only staging on the frontend**: invoke
   `huggingface_hub.snapshot_download(repo_id="xlm-roberta-base",
   revision="<sha>", cache_dir="${HF_HOME}")` and
   `snapshot_download(repo_id="sat-3l-sm",
   revision="<sha>", cache_dir="${HF_HOME}")`. This only fetches
   files; no model is loaded.
2. **rsync an already-cached snapshot**: if `HF_HOME` was
   previously populated (for example on a prior smoke node), copy
   the directory tree to the new persistent root. Files only;
   source code never travels this path.

The preflight script asserts `HF_HUB_OFFLINE=1` is in effect so
no network fetch is attempted at runtime.

`HF_HOME` must remain under the approved persistent remote root;
`/tmp`, `/var/tmp`, and `/dev/shm` are explicitly rejected by
`run_gpu_smoke.sh` as forbidden ephemeral storage.

## Persistent storage layout

The smoke runs under the operator's persistent home directory
on the Nancy frontend and compute nodes. `/srv/storage` is not
assumed to expose a usable group storage for this account — the
operator's home is the canonical persistent root.

```bash
G5K_LOGIN="<grid5000-login>"
SOURCE_COMMIT="<exact-40-character-commit-sha>"
REMOTE_PERSISTENT_ROOT="/home/${G5K_LOGIN}/g5k-smoke/${SOURCE_COMMIT}"
REPO_ROOT=${REMOTE_PERSISTENT_ROOT}/repo
HF_HOME=${REMOTE_PERSISTENT_ROOT}/hf_home
LOG_ROOT=${REMOTE_PERSISTENT_ROOT}/logs

# Export every persistent variable before any downstream command
# (download-only staging, OAR submission, smoke run) so the values
# are reliably substituted regardless of the shell context.
export REMOTE_PERSISTENT_ROOT
export REPO_ROOT
export HF_HOME
export LOG_ROOT
```

On the remote site, pin the checkout to the exact commit:

```bash
git checkout --detach "${SOURCE_COMMIT}"
test "$(git rev-parse HEAD)" = "${SOURCE_COMMIT}"
```

The locked Python environment lives **inside the checkout**, at
`${REPO_ROOT}/.venv`; the smoke uses
`${REPO_ROOT}/.venv/bin/python`. No standalone `.venv` is kept
under `${REMOTE_PERSISTENT_ROOT}/.venv`.

Before creating these directories, re-check quota headroom:

```bash
quota -s
df -h "/home/${G5K_LOGIN}"
```

If hard-quota remaining (`df` + soft/hard) is below **15 GB**,
abort and do not proceed.

## Cache staging plan (download-only, no Mac computation)

This is the exact plan that the smoke expects to find on the
remote compute node. No part of it runs on the Mac or on the
Grid'5000 frontend inference code path; only file fetches and
file copies.

### Required Hub assets

The smoke depends on exactly two Hugging Face assets:

1. **`segment-any-text/sat-3l-sm`** — the `wtpsplit` SaT weights
   that `SaTSentenceSegmenter(model_name="sat-3l-sm")` resolves
   through wtpsplit's default `hub_prefix="segment-any-text"`.
2. **`facebookAI/xlm-roberta-base`** — the XLM-R tokenizer that
   `SaT(tokenizer_name_or_path="facebookAI/xlm-roberta-base")`
   loads by default in wtpsplit 2.2.1.

Both repos are public; neither requires an `HF_TOKEN`. A token
must **never** be configured or logged during staging.

### Immutable revisions

Pin the exact commit SHA for each repo on the remote side before
running the smoke. Both SHAs below were resolved metadata-only
against `huggingface.co/api/models/.../tree/<sha>`; no download
or model construction occurred during resolution.

| Hub repo | Public commit SHA |
| --- | --- |
| `segment-any-text/sat-3l-sm` | `137da054051ad9f1eac42025f758db4ac9f22535` |
| `facebookAI/xlm-roberta-base` | `e73636d4f797dec63c3081bb6ed5c7b0bb3f2089` |

Record both SHAs in the operator's run log next to the commit-B
SHA from the [Commit-based remote
reproducibility](#commit-based-remote-reproducibility) section.

### Selective `allow_patterns` (no framework weight dumps)

For PyTorch SaT inference only the following files are required,
**no others**:

For `segment-any-text/sat-3l-sm@137da054051ad9f1eac42025f758db4ac9f22535`:

- `config.json`
- `model.safetensors`

For `facebookAI/xlm-roberta-base@e73636d4f797dec63c3081bb6ed5c7b0bb3f2089`:

- `config.json`
- `tokenizer.json`
- `tokenizer_config.json`
- `sentencepiece.bpe.model`

(`special_tokens_map.json` is also accepted when present; this
revision does not include it.) ONNX, Flax, TensorFlow,
`pytorch_model.bin`, and the XLM-R `model.safetensors` are
**not** required — the SaT smoke loads only the tokenizer from
the XLM-R base and the PyTorch backbone from `sat-3l-sm`.

### Expected cache size (selective)

| Repo | Selective footprint |
| --- | --- |
| `segment-any-text/sat-3l-sm` (config + safetensors) | **≈ 815 MiB** |
| `facebookAI/xlm-roberta-base` (4 tokenizer files) | **≈ 13.5 MiB** |
| **Selective total** | **≈ 828 MiB ≈ 0.81 GB** |

Re-verify after staging with `du -sh`. If either cache directory
is materially smaller than expected, abort (the snapshot is

### Cache location and download procedure (no inference, no model construction)

The persistent variables must be exported before any staging
heredoc runs:

```bash
export REMOTE_PERSISTENT_ROOT
export REPO_ROOT
export HF_HOME
export LOG_ROOT
```

When `HF_HOME` is set, the Hub stack searches `${HF_HOME}/hub`.
Download only -- no model construction, no inference -- using the
authenticated-free `huggingface_hub.snapshot_download` API:

```bash
"${REPO_ROOT}/.venv/bin/python" - <<'PYEOF'
import os
from huggingface_hub import snapshot_download
cache_dir = os.path.join(os.environ["HF_HOME"], "hub")
snapshot_download(
    repo_id="segment-any-text/sat-3l-sm",
    revision="137da054051ad9f1eac42025f758db4ac9f22535",
    cache_dir=cache_dir,
    allow_patterns=["config.json", "model.safetensors"],
)
snapshot_download(
    repo_id="facebookAI/xlm-roberta-base",
    revision="e73636d4f797dec63c3081bb6ed5c7b0bb3f2089",
    cache_dir=cache_dir,
    allow_patterns=[
        "config.json",
        "tokenizer.json",
        "tokenizer_config.json",
        "sentencepiece.bpe.model",
        "special_tokens_map.json",  # accepted when present; no-op if missing
    ],
)
PYEOF
```

**Forbidden during staging:**

- No `SaT(...)`, `SaTSentenceSegmenter(...)`,
  `AutoModelForTokenClassification.from_pretrained(...)`, or any
  other model-construction call.
- No `split(...)`, `split_batch(...)`, or any inference call.
- No `torch` import (the staging helper only imports
  `huggingface_hub.snapshot_download`).
- No system `python3` (use only `${REPO_ROOT}/.venv/bin/python`).
- No `HF_TOKEN` export.

### Bind offline name resolution to the verified snapshot

The smoke constructs models by **repository name** without a
revision. A cache populated only by exact-SHA downloads does not
have an entry under `refs/main`, so the offline lookup would
miss the snapshot. After the staged download:

```bash
SAT_REFS_DIR="${HF_HOME}/hub/models--segment-any-text--sat-3l-sm/refs"
XLM_REFS_DIR="${HF_HOME}/hub/models--facebookAI--xlm-roberta-base/refs"

# Create the refs parent directories before writing refs/main.
# Without this, the heredoc would fail on a freshly populated
# hub cache. Mode 0700 is enforced so a downstream reader of
# the cache cannot accidentally depend on these hand-written
# refs for unrelated reasons.
mkdir -p -m 0700 "${SAT_REFS_DIR}" "${XLM_REFS_DIR}"

# Defense in depth: a restrictive umask for the printf below.
umask 077
printf '137da054051ad9f1eac42025f758db4ac9f22535\n' > "${SAT_REFS_DIR}/main"
printf 'e73636d4f797dec63c3081bb6ed5c7b0bb3f2089\n'  > "${XLM_REFS_DIR}/main"

# Pin the refs to the exact SHA downloaded. No unrelated cache
# entry is modified.
SAT_REFS="${SAT_REFS_DIR}/main"
XLM_REFS="${XLM_REFS_DIR}/main"
test "$(cat "${SAT_REFS}")" = "137da054051ad9f1eac42025f758db4ac9f22535"
test "$(cat "${XLM_REFS}")" = "e73636d4f797dec63c3081bb6ed5c7b0bb3f2089"
```

These refs bind the offline default-name lookup (`sat-3l-sm`
or `facebookAI/xlm-roberta-base`) to the already-verified
immutable snapshots. No unrelated cache entry is modified.

### Reusing an already-cached snapshot

If a prior smoke populated `${HF_HOME}` (for example on a
previous OAR allocation), the snapshot can be reused by `rsync`
between persistent directories. Source code and lockfiles must
not travel this path:

```bash
rsync -a --delete \
    "${OLD_HF_HOME}/" \
    "${NEW_HF_HOME}/"
chmod 0700 "${NEW_HF_HOME}"
```

After this, the OAR payload's preflight JSON reports the exact
`device_0_name` and the smoke constructs the model with
`device="cuda"` from the cached assets only. `HF_HUB_OFFLINE=1`
and `TRANSFORMERS_OFFLINE=1` are forced by the payload, so even
a transient network failure during the allocation cannot fall
back to a runtime download.

## Environment setup from the locked dependencies

Environment installation must happen **on the Nancy frontend
before reserving GPU walltime**. `oarsub -I` is interactive —
once it starts the 30-minute wall-clock is in effect — so the
Python environment is prepared beforehand.

### One-time per-phase command set

```bash
# From the operator's Grid'5000 login session (frontend or nancy):
cd "${REPO_ROOT}"
test "$(git rev-parse HEAD)" = "${SOURCE_COMMIT}"
test -z "$(git status --porcelain)"
export PATH="/home/${G5K_LOGIN}/.local/bin:${PATH}"

# Use a job-specific temporary uv cache so cleanup is exact. The
# variable is exported (not command-scoped) so we can validate
# the exact /tmp/uv-cache-g5k-<USER>-<PID> form before deletion.
UV_CACHE_DIR="/tmp/uv-cache-g5k-${USER}-$$"
export UV_CACHE_DIR
uv sync --locked --extra segmentation

# Verify with the locked interpreter -- no imports beyond
# importlib.metadata; no SaT construction.
"${REPO_ROOT}/.venv/bin/python" -c \
    "import importlib.metadata as m; print(m.version('torch')); print(m.version('wtpsplit'))"

# After successful installation: remove the exact UV cache we
# created. Match the /tmp/uv-cache-g5k-<USER>-* prefix exactly
# before deletion; never use a broad glob like /tmp/uv-cache-*.
case "${UV_CACHE_DIR}" in
    "/tmp/uv-cache-g5k-${USER}-"*) ;;
    *) echo "refusing unsafe UV cache cleanup" >&2; exit 1 ;;
esac
test -n "${UV_CACHE_DIR}"
rm -rf -- "${UV_CACHE_DIR}"
unset UV_CACHE_DIR
```

The locked interpreter resolves to
`${REPO_ROOT}/.venv/bin/python` and contains
`wtpsplit==2.2.1` plus the locked CUDA-enabled `torch`. The
segmenter refuses any other `wtpsplit` version at runtime, so a
lockfile drift would fail loudly rather than silently.

## Smoke-test command

The canonical submission path is the **non-interactive batch
form** documented above under *Non-interactive batch
submission*. Do not submit a new interactive `-I` job for the
smoke: the batch wrapper is the supported entrypoint and the
operator does not need to hold a TTY-bridge for the duration of
the run.

If an existing interactive allocation is already open (for
example, a human-held debugging session), the smoke harness may
be invoked manually inside that allocation. This is a
debugging-only path; it is **not** the canonical submission
form. Use the batch wrapper for any production smoke run:

```bash
# Inside an already-open interactive allocation only.
# For production, prefer the batch wrapper documented above.

# OAR_JOB_ID is scheduler-owned. DO NOT overwrite it. The
# payload does NOT require CUDA_VISIBLE_DEVICES; the preflight
# proves GPU scoping via torch.cuda.device_count() == 1. If
# the scheduler happens to set CUDA_VISIBLE_DEVICES, the
# harness inherits it unchanged.
: "${OAR_JOB_ID:?scheduler must set OAR_JOB_ID}"

export REPO_ROOT=<as defined under Persistent storage layout>
export HF_HOME=<as defined under Persistent storage layout>
export LOG_ROOT=<as defined under Persistent storage layout>
export EXPECTED_SOURCE_COMMIT=<exact 40-char SHA of the commit you intend to smoke>

# Verify the checkout matches the intended commit BEFORE running.
test "$(git -C "${REPO_ROOT}" rev-parse HEAD)" = "${EXPECTED_SOURCE_COMMIT}"
test -z "$(git -C "${REPO_ROOT}" status --porcelain)"

export SMOKE_LOG_DIR="${LOG_ROOT}/${OAR_JOB_ID}"

# Job-specific log directory: must not pre-exist. Create with
# mode 0700. Use mkdir (not mkdir -p) so a stale directory fails
# loudly instead of silently inheriting permissions.
test ! -e "${SMOKE_LOG_DIR}"
mkdir -m 0700 "${SMOKE_LOG_DIR}"
for f in gpu_preflight.json smoke_result.json run_metadata.json \
         smoke.stdout.log smoke.stderr.log; do
    if [ -e "${SMOKE_LOG_DIR}/${f}" ]; then
        echo "refusing to overwrite ${f}" >&2
        exit 1
    fi
done

# Run smoke once; capture exit code separately. tee is
# deliberately avoided so exit-status is unambiguous.
set +e
bash "${REPO_ROOT}/scripts/grid5000/run_gpu_smoke.sh" \
    >"${SMOKE_LOG_DIR}/smoke.stdout.log" \
    2>"${SMOKE_LOG_DIR}/smoke.stderr.log"
smoke_rc=$?
set -e

# Persist the exit code immediately. Do not re-run the smoke on
# failure; the recorded exit code drives downstream triage.
printf '%s\n' "${smoke_rc}" > "${SMOKE_LOG_DIR}/smoke.exit_code"
```

The batch wrapper `run_gpu_smoke_job.sh` performs every step
above on the operator's behalf, plus the strict path
canonicalisation, `OAR_JOB_ID` numeric validation, exact
four-positional-argument check, and post-success contract that
requires exactly six direct entries (each an expected regular
file, mode 0600) inside a 0700 directory, documented in
*Non-interactive batch submission*. The manual snippet above is
retained only for debugging an already-open allocation; it does
not replace the batch wrapper.

`tee` is intentionally not used — it complicates exit-status
handling — but the exit code is recorded separately so the
operator can detect failure after the fact without re-running.

The shell `run_gpu_smoke.sh` itself:

1. Refuses to start without `OAR_JOB_ID`. It does NOT require
   `CUDA_VISIBLE_DEVICES`; the preflight's
   `torch.cuda.device_count() == 1` check is the authoritative
   runtime proof of GPU scoping (see
   [the Grid'5000 GPU reservation
   documentation](https://www.grid5000.fr/w/GPUs_on_Grid5000)).
2. Refuses to overwrite any pre-existing `gpu_preflight.json`,
   `smoke_result.json`, or `run_metadata.json`.
3. Refuses to start unless `command -v git` succeeds,
   `EXPECTED_SOURCE_COMMIT` is set and matches
   `^[0-9a-f]{40}$`, the on-machine `git -C "${REPO_ROOT}"
   rev-parse HEAD` equals `EXPECTED_SOURCE_COMMIT`, and
   `git -C "${REPO_ROOT}" status --porcelain` is empty.
   This prevents accidentally running from a different revision
   or a dirty checkout.
4. Runs `scripts/grid5000/gpu_preflight.py`, asserts Linux, the
   OAR allocation, `torch.cuda.is_available()`, and exactly one
   visible CUDA device.
5. Validates the preflight JSON contract and atomically installs
   `gpu_preflight.json` (`os.link`, no `mv`, no overwrite).
6. Writes deterministic `run_metadata.json` (mode 0600, atomic)
   with `source_commit`, `model_name`, `model_revision`,
   `tokenizer_name`, `tokenizer_revision`, `oar_job_id`, and
   `hostname` only.
7. Constructs `SaTSentenceSegmenter(model_name="sat-3l-sm",
   device="cuda")` — **explicit CUDA, never `auto`** — and runs
   inference once on the small multilingual batch.
8. Asserts `resolved_device == "cuda"`, that the output group
   count equals the input count, and that every group has at
   least one non-blank sentence.
9. Writes a compact JSON result (`resolved_device`,
   `model_name`, `input_count`, `sentence_counts`,
   `elapsed_seconds`, `torch_version`,
   `torch_cuda_runtime_version`, `cuda_device_name`) to
   `${SMOKE_LOG_DIR}/smoke_result.json`.

Final `${SMOKE_LOG_DIR}` layout (6 files):

- `gpu_preflight.json`
- `run_metadata.json`
- `smoke_result.json`
- `smoke.stdout.log`
- `smoke.stderr.log`
- `smoke.exit_code`

## Phase 9L-B: full resumable build (pinned local input)

Phase 9L-A added per-shard resumable checkpoints; Phase 9L-B
extends the Grid'5000 launcher family from the small smoke to
the full resumable build of the public
`NoeFlandre/osm-polygon-wikidata-only` dataset
(resolve the upstream `main` revision metadata-only against the
Hub API before staging; the resolved 40-character lowercase hex
SHA is the canonical `INPUT_REVISION`),
**291 processable shards** at the time of writing.

The build launcher mirrors the safety contract of the smoke
launcher (frontend adapter + compute-node wrapper +
compute-node payload), adds three positional arguments, and
tightens the path and revision contracts:

| Phase | Script pair | Positional args |
| --- | --- | --- |
| 9D/9F | `submit_gpu_smoke.sh` + `run_gpu_smoke_job.sh` | `REPO_ROOT HF_HOME LOG_ROOT EXPECTED_SOURCE_COMMIT` |
| 9L-B / 9M-B | `submit_gpu_build.sh` + `run_gpu_build_job.sh` | `REPO_ROOT HF_HOME LOG_ROOT INPUT_ROOT WORK_DIR OUTPUT_DIR EXPECTED_SOURCE_COMMIT INPUT_REVISION BATCH_SIZE` |

The nine positional arguments are positional in the order
above; do **not** try to pass them as `--flag value` pairs (the
frontend adapter validates `$# == 9` and the wrapper does too).
The ninth argument `BATCH_SIZE` is the positive-integer batch
size passed to the build CLI's `--batch-size` flag; the launcher
rejects empty, boolean, decimal, leading-zero, and non-integer
values at submission time. The 128 default that applies to
ordinary CLI use is **not** used by the launcher; the operator
must pass an explicit value (e.g., `128` for the documented
default behaviour, or `16` for the small-batch diagnostic
benchmark).

### Pinned local-input mode

The build launcher is **local-input only**. It invokes the
existing public CLI in local mode:

```text
${REPO_ROOT}/.venv/bin/osm-polygon-sentence-relevance \
    --input-root "${INPUT_ROOT}" \
    --input-source-dataset-id "NoeFlandre/osm-polygon-wikidata-only" \
    --input-dataset-revision "${INPUT_REVISION}" \
    --output-dir "${OUTPUT_DIR}" \
    --pipeline-version 0.1.0 \
    --device cuda \
    --batch-size "${BATCH_SIZE}" \
    --sat-model sat-3l-sm \
    --work-dir "${WORK_DIR}" \
    --source-commit "${EXPECTED_SOURCE_COMMIT}"
```

The launcher **must not** be given `--input-dataset-id` (Hub
mode); `--input-root` and `--input-source-dataset-id` together
identify the immutable local snapshot and the upstream Hub
dataset for provenance without triggering any network fetch.
`--device cuda` is **explicit only**: no `auto`, `cpu`, or `mps`
fallback. No publishing flags (`--publish-dataset-id`,
`--publish-revision`, `--publish-commit-message`) and no
destructive re-rendering flag may be added to the invocation.

### INPUT_REVISION: 40 hex chars, never `main`

`INPUT_REVISION` is the immutable 40-character lowercase hex
SHA of the upstream Hub snapshot (resolve metadata-only against
`huggingface.co/api/datasets/NoeFlandre/osm-polygon-wikidata-only/tree/<sha>`
before staging). The submission adapter and the compute-node
wrapper both reject `main`, empty strings, short SHAs, and
uppercase hex with a non-zero exit.

### Path relationship contract

All five persistent paths (`REPO_ROOT`, `HF_HOME`, `LOG_ROOT`,
`INPUT_ROOT`, `WORK_DIR`, `OUTPUT_DIR`) must be:

- **absolute**, no `..`, `//`, or trailing `/.`;
- **canonical real directories** (no symlinks at any operator-
  controlled path component, verified with `cd -- && pwd -P`);
- **persistent** — `/tmp`, `/var/tmp`, `/dev/shm` are
  explicitly rejected;
- **disjoint** — `INPUT_ROOT`, `WORK_DIR`, `OUTPUT_DIR`, and
  `REPO_ROOT` must not overlap. `_check_overlap` rejects any
  pair where one path is the strict prefix of the other at a
  path-segment boundary.

`INPUT_ROOT` and `LOG_ROOT` must already exist. `WORK_DIR` may
already exist (resume case); the wrapper accepts it, runs the
git HEAD check, and lets the pipeline resume from any valid
checkpoints already present. `OUTPUT_DIR` must **not** exist
(fresh build only) — the wrapper refuses to reuse an existing
`OUTPUT_DIR` and the payload creates it fresh.

### Resume contract (Phase 9L-A + 9L-B)

A failed or walltime-terminated OAR job leaves `WORK_DIR`
untouched. The build payload:

- does **not** delete or quarantine any pre-existing checkpoint
  on failure;
- propagates the real CLI exit code through `build.exit_code`;
- never retries on CPU, MPS, or `auto`;
- never falls back to a re-download or Hub fetch.

A subsequent OAR submission with the **same `WORK_DIR`** and
**same `INPUT_REVISION`** resumes from the last valid
checkpoint; the launcher never modifies `WORK_DIR` between
runs.

### Walltime (proposed, pending benchmark)

The submission adapter requests `gpu=1,walltime=12:00:00` on
the production queue. This is a **proposed upper bound**,
pending measurement of representative-shard throughput on
the chosen Nancy GPU family. The value is **not** yet
validated as sufficient for the full 291-shard build;
operators are expected to measure one or two representative
shards and adjust the walltime accordingly.

## Phase 9M: diagnostic amendment (exception chain, batch size, input audit, site search)

Phase 9M is a **diagnostic amendment** that surfaces three
operational blockers without changing the build contract:

- **9M-A** — surface exception causes publicly. The CLI's outer
  `except` now renders a bounded, cycle-safe, traceback-free
  `__cause__` chain (`Error: ...` followed by one
  `Caused by: <Type>: <message>` line per hop, with a
  `Caused by: <truncated>` tail when the chain is deeper than
  the bound). This makes the underlying CUDA OOM, weight-load
  mismatch, or join-integrity failure visible in the build
  log on stderr with exit code 1 — no traceback frames, no
  local paths.
- **9M-B** — make `--batch-size` explicit at the launcher.
  The ninth positional arg `BATCH_SIZE` is validated at three
  layers (frontend adapter, compute-node wrapper, payload)
  and substituted as `--batch-size "${BATCH_SIZE}"`. The
  documented CLI default of 128 remains the value passed
  for ordinary CLI use; the remote launcher does not default.
- **9M-C / 9M-D** — pure offline audit + read-only site
  search; no full build, no publishing, no data mutation.

The amendment is **uncommitted-for-review** by design. After
approval and push, the next operational benchmark on a known
small shard will exercise both fixes: the next bayern run at
`BATCH_SIZE=128` will print the formatted exception chain
instead of `Error: <bare>`, immediately revealing whether the
remaining blocker is a CUDA OOM, a model weight load error, or
something else. If and only if it is a CUDA OOM, the planned
small-batch benchmark at `BATCH_SIZE=16` is authorised using
the same persistent paths.

### No-publishing boundary

The launcher family is for the **build only**. It explicitly
forbids, in the source and in the static tests:

- any `--publish-*` flag (dataset id, revision, commit
  message);
- any `--overwrite` flag on the build CLI invocation;
- any Hub re-fetch (`HF_HUB_OFFLINE=1`,
  `TRANSFORMERS_OFFLINE=1` are forced);
- any model construction on the Mac, on the Grid'5000 frontend,
  on CPU, on MPS, or on `auto`;
- any rsync of source code or lockfiles;
- any `git commit`, `git push`, `git checkout`, `git fetch`,
  `git clone`, `git reset`, `git pull`, `git stash`,
  `git cherry-pick`, or `git rebase` inside the payload.

The build is the *boundary*: upstream datasets are immutable,
the input snapshot is immutable, the source commit is pinned,
and the build output is the only mutable artefact.
